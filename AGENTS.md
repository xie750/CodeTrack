## 执行框架原则

- 构建可端到端演示的薄垂直切片。
- 让仓库内的 Markdown、Schema、种子数据、测试和 UI 状态对代理保持清晰可读。
- 优先使用稳定契约，而不是大型提示词。
- 区分产品文档、页面规格、API 契约、模拟数据、AI 工作流和验收检查。
- 固定学习流程使用工作流，开放式帮助使用代理。
- 清晰标注哪些内容是真实的、模拟的、基于规则的或计划中的。
- 面向学生展示的每个 AI 输出，在相关时都必须暴露来源、置信度和下一步动作。
- 避免一个巨大的 AI 聊天框；应将系统建模为具有明确输入和输出的小型代理。

## 当前 P0 场景

第一个可运行执行框架应支持以下学生助学闭环：

```text
学习首页
-> 已分配课程任务
-> 编码工作区
-> 沙箱执行
-> 带引用的 AI 诊断
-> 渐进式提示
-> 重新提交
-> 学习总结
-> 学习者画像更新
-> 保存学习产物
-> 下一步推荐
```

同时支持一个轻量级自学闭环：

```text
自学
-> 选择知识点
-> 生成讲解 / 笔记 / 卡片 / 思维导图 / PPT 大纲
-> 保存产物
-> 更新学习者画像
```

## 当前学生页面

- 学习首页。
- 课程任务。
- 任务工作区。
- 自学。
- AI 导师。
- 我的资料库。
- 学习者画像摘要，应展示在首页或个人资料区域内，而不是作为一级导航。

## 代理角色

使用 `Dorc/第二阶段/07_agent_harness_design.md` 中的这些角色：

- 学习导航代理。
- 代码诊断代理。
- 渐进式提示代理。
- 概念导师代理。
- 学习产物生成代理。
- 学习者画像代理。
- 引用守卫代理。

首版实现可以对大多数代理进行模拟或规则兜底，但它们的输入和输出应保持稳定。

## 工程技术栈

- 后端：FastAPI、SQLAlchemy、Alembic、pytest。
- 前端：React、TypeScript、Vite、Ant Design、Monaco Editor、lucide-react。
- 沙箱：使用现有 C++ 执行路径。

## UI 规则

- 遵循 `Dorc/第二阶段/04_ui_ux_design.md`。
- 使用现代 SaaS / AI 工作区布局。
- 桌面端：左侧导航、顶部工具栏、主工作区、可选右侧 AI/上下文面板。
- 移动端：顶部标题、内容流、底部导航最多五项。
- 不要把营销落地页作为主要体验。
- 不要用 emoji 作为结构性图标。
- 卡片应克制、易读且功能明确。

## 边界

除非明确要求，否则不要在第一版学生助学执行框架中加入以下内容：

- 助教教学看板。
- 科研助手场景。
- 多学科切换。
- 真实视频生成。
- 完整 PPT 二进制导出。
- 完整 LMS 或教务管理系统。
- 游戏化、排行榜或社交社区。
