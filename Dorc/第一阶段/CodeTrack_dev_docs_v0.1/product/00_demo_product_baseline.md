# Demo V0.1 产品基线

**状态：首个 Demo 开发期间冻结**

本文档把第一阶段分散在范围、业务流、页面、数据、API 和测试文档中的关键产品规则收束成一页基线。编码、设计和验收时若出现冲突，以本文档和 `99_open_decisions.md` 为第一优先级。

## 1. 产品目标

首个 Demo 验证一条真实学习闭环：

```text
学生提交单链表删除代码
→ 系统真实编译和测试
→ 基于失败证据与课程资料生成诊断
→ 给出不泄露完整答案的渐进提示
→ 学生修改并重新提交
→ 测试通过
→ 生成可追溯能力证据
→ 教师查看过程时间线
```

Demo 成功不以页面数量、视觉精度或功能丰富度判断，而以这条链路是否真实、稳定、可复核判断。

## 2. 首个任务

- 课程：数据结构。
- 语言：C++。
- 任务：单链表指定位置节点删除。
- 首个标准错误：删除位置为 `0` 时，没有更新链表头指针。
- 能力点：`LINKED_LIST_BOUNDARY_HANDLING`，即链表边界处理能力。

固定函数接口：

```cpp
ListNode* deleteAt(ListNode* head, int position);
```

学生只实现指定函数。平台负责提供 `ListNode`、测试驱动和编译模板。

## 3. 页面范围

学生端首个 Demo 必须包含：

- 任务列表；
- 任务工作台；
- 代码编辑器；
- 提交与执行状态；
- 编译和测试结果；
- AI 诊断与课程引用；
- 渐进提示；
- 版本历史；
- 完成总结。

教师端首个 Demo 必须包含：

- 提交列表；
- 单个提交时间线；
- 测试失败、诊断、提示记录和最终结果。

不做复杂统计大屏、完整任务创建器、管理员后台、游戏化和多课程切换。

## 4. 信息分区

页面必须清晰区分三类信息：

- 工具事实：编译、运行、测试和沙箱状态；
- 课程依据：RAG 检索到的知识来源；
- AI 分析：错因解释、置信度、提示和复核标记。

AI 文案不得覆盖工具事实。模型失败时，学生仍然能查看编译和测试结果，并继续修改和重新提交。

## 5. 提示规则

默认支持三级提示：

- 一级：失败现象和排查方向；
- 二级：知识点和可能出错区域；
- 三级：修正步骤或伪代码。

一级提示禁止：

- 给出完整修复代码；
- 直接输出完整正确语句；
- 输出完整函数。

完整参考答案是独立操作，不属于默认提示层级。查看参考答案后，只记录任务完成，不产生强能力掌握证据。

## 6. 能力证据

能力证据由规则服务生成，不由大模型直接决定。

证据强度规则：

- 独立通过关键测试：强正向证据；
- 使用一级提示后修复通过：中等正向证据；
- 使用二级或三级提示后通过：弱正向证据；
- 查看参考答案后通过：中性完成证据；
- 多次重复同类边界错误：负向证据。

能力状态使用：

- 待观察；
- 需要支持；
- 初步掌握；
- 基本掌握。

学生和教师都应能展开查看证据来源。

## 7. 验收底线

首个 Demo 必须满足：

- 错误代码能真实提交；
- 删除头节点测试失败；
- 正确代码或修复后代码通过全部必要测试；
- 每次重新提交生成新版本；
- 旧版本和原始测试结果不可覆盖；
- AI 输出通过 Schema 校验；
- 诊断引用真实测试证据和真实课程来源；
- 一级提示人工检查无完整答案泄露；
- 能力证据能回溯到版本、测试和提示记录；
- 教师端能看到完整时间线。

## 8. 暂不处理

以下事项不进入当前产品验收：

- 部署环境；
- 域名和 HTTPS；
- 多校多租户；
- 大规模并发；
- 多编程语言；
- 完整能力评分算法；
- 完整课程市场；
- 高保真视觉和复杂动效。
